**Step 2: Create `packages/api/Dockerfile` (per D-01, D-02, D-03).**

Four-stage multi-stage build, built from repo root context (`docker build -f packages/api/Dockerfile .`):

```dockerfile
# Stage 1: Prune the monorepo to API-only subset
FROM node:22-slim AS pruner
RUN npm install -g turbo@2
WORKDIR /app
COPY . .
RUN turbo prune @seatkit/api --docker

# Stage 2: Install production dependencies (cached when lockfile unchanged)
FROM node:22-slim AS installer
RUN npm install -g pnpm@9.12.3
WORKDIR /app
COPY --from=pruner /app/out/json/ .
COPY --from=pruner /app/out/pnpm-lock.yaml ./pnpm-lock.yaml
COPY --from=pruner /app/out/pnpm-workspace.yaml ./pnpm-workspace.yaml
RUN pnpm install --frozen-lockfile

# Stage 3: Build the API
FROM installer AS builder
COPY --from=pruner /app/out/full/ .
RUN pnpm build --filter=@seatkit/api

# Stage 4: Production runner
FROM node:22-slim AS runner
RUN addgroup --system seatkit && adduser --system --ingroup seatkit seatkit
WORKDIR /app

# Copy built output and runtime dependencies
COPY --from=builder /app/packages/api/dist ./dist
COPY --from=builder /app/packages/api/package.json ./package.json
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/packages/api/node_modules ./packages/api/node_modules 2>/dev/null || true
# Copy migration files — migrate.ts reads from ./src/db/migrations
COPY --from=builder /app/packages/api/src/db/migrations ./src/db/migrations

USER seatkit

ENV NODE_ENV=production
ENV PORT=3001
EXPOSE 3001
CMD ["node", "dist/index.js"]
```

Critical notes for the executor:
- The API's tsup build externalizes all node_modules (ESM bare imports). Both root `node_modules/` and `packages/api/node_modules/` are needed because pnpm hoists to root and symlinks from package dir.
- `src/db/migrations/` must be copied because `migrate.ts` reads from `./src/db/migrations` at runtime.
- The `2>/dev/null || true` on the api node_modules COPY handles the case where pnpm fully hoists (no local node_modules).
- Do NOT copy `.env` files — secrets are mounted at runtime via Cloud Run (D-18).
- turbo v2.5.8 uses positional args: `turbo prune @seatkit/api --docker` (NOT `--scope`).

**Step 2: Create `packages/web/Dockerfile` (per D-01, D-02, D-03, D-04, D-20).**

Four-stage multi-stage build with `NEXT_PUBLIC_API_URL` as a build-time arg:

```dockerfile
# Stage 1: Prune the monorepo to web-only subset
FROM node:22-slim AS pruner
RUN npm install -g turbo@2
WORKDIR /app
COPY . .
RUN turbo prune @seatkit/web --docker

# Stage 2: Install dependencies (cached when lockfile unchanged)
FROM node:22-slim AS installer
RUN npm install -g pnpm@9.12.3
WORKDIR /app
COPY --from=pruner /app/out/json/ .
COPY --from=pruner /app/out/pnpm-lock.yaml ./pnpm-lock.yaml
COPY --from=pruner /app/out/pnpm-workspace.yaml ./pnpm-workspace.yaml
RUN pnpm install --frozen-lockfile

# Stage 3: Build with NEXT_PUBLIC_API_URL baked in (per D-20)
FROM installer AS builder
ARG NEXT_PUBLIC_API_URL
ENV NEXT_PUBLIC_API_URL=${NEXT_PUBLIC_API_URL}
COPY --from=pruner /app/out/full/ .
RUN pnpm build --filter=@seatkit/web

# Stage 4: Production runner (standalone = no node_modules needed)
FROM node:22-slim AS runner
RUN addgroup --system seatkit && adduser --system --ingroup seatkit seatkit
WORKDIR /app

# Copy standalone server output
# In a pnpm monorepo, standalone output preserves the packages/web/ directory structure
COPY --from=builder /app/packages/web/.next/standalone ./
COPY --from=builder /app/packages/web/.next/static ./packages/web/.next/static
COPY --from=builder /app/packages/web/public ./packages/web/public

USER seatkit

ENV NODE_ENV=production
ENV PORT=3000
EXPOSE 3000

# In monorepo standalone mode, server.js is at packages/web/server.js
CMD ["node", "packages/web/server.js"]
```

Critical notes for the executor:
- `NEXT_PUBLIC_API_URL` is a build-time arg (D-20) — it gets baked into the JS bundle during `next build`. The deploy workflow passes `--build-arg NEXT_PUBLIC_API_URL=https://api-staging.seatkit.dev`.
- Next.js standalone output in a monorepo preserves the directory hierarchy. The `server.js` entry point is at `packages/web/server.js` within the standalone output, NOT at the root. The COPY of `.next/standalone ./` places the full tree at `/app/`, so `server.js` ends up at `/app/packages/web/server.js`.
- `.next/static` and `public/` are NOT included in standalone output and must be copied separately. Their paths must match what the standalone server expects: `packages/web/.next/static` and `packages/web/public`.
- If the monorepo standalone path assumption is wrong (A1 from research), the container will fail to start with "Cannot find module" — the executor should verify the actual path by inspecting the build output if possible.
- Do NOT copy `.env` or secret files.
