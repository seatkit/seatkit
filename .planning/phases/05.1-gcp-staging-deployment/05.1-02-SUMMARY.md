## Decisions Made
- Per-secret `roles/secretmanager.secretAccessor` IAM bindings (4 separate resources) instead of a single project-wide binding -- follows least-privilege principle per threat model T-05.1-04
- `lifecycle.ignore_changes` on `template[0].containers[0].image` prevents Terraform from reverting CI/CD-deployed images -- CI manages images, Terraform manages service config
- Cloud Run v1 `google_cloud_run_domain_mapping` resource used for custom domains because Cloud Run v2 API does not support domain mappings -- this is the standard Terraform pattern

## User Setup Required

Before running `terraform apply`, the developer must:
1. Replace `REPLACE_WITH_PROJECT_ID` in `infra/staging/terraform.tfvars` with the actual GCP project ID
2. Replace `REPLACE_WITH_DNS_ZONE_NAME` with the Cloud DNS managed zone name for seatkit.dev
3. Create the GCS state bucket manually: `gsutil mb gs://seatkit-terraform-state`
4. Create the 4 Secret Manager secrets and add their values (DATABASE_URL, BETTER_AUTH_SECRET, ADMIN_EMAIL, ADMIN_PASSWORD)
5. Run `terraform init && terraform plan && terraform apply` from `infra/staging/`
