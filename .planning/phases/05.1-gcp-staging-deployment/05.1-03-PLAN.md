  <action>
Create `.github/workflows/deploy.yml` with the following structure and content.

**Trigger configuration (D-13, D-14, D-17):**
- `on.workflow_run`: triggers on `workflows: ["CI"]`, `types: [completed]`, `branches: [main]`
- `on.workflow_dispatch`: has an `inputs.service` of `type: choice` with options `api`, `web`, `both`

**Concurrency:**
- `group: deploy-${{ github.ref }}`
- `cancel-in-progress: false` (never cancel an in-progress deploy)

**Environment variables:**
- `REGION: europe-west1` (D-12)
- `GAR_REGISTRY: europe-west1-docker.pkg.dev/${{ secrets.GCP_PROJECT_ID }}/seatkit` (D-05, D-12)

**Job: deploy-api (D-16 — runs in parallel with deploy-web, no `needs`):**

The `if` condition must handle both trigger types:
```yaml
if: >
  (github.event_name == 'workflow_dispatch' &&
   (github.event.inputs.service == 'api' || github.event.inputs.service == 'both')) ||
  (github.event_name == 'workflow_run' &&
   github.event.workflow_run.conclusion == 'success')
```

Steps:
1. `actions/checkout@v4`
2. `google-github-actions/auth@v2` with `credentials_json: ${{ secrets.GCP_SERVICE_ACCOUNT_KEY }}`
3. `google-github-actions/setup-gcloud@v2`
4. Configure Docker auth: `gcloud auth configure-docker europe-west1-docker.pkg.dev --quiet`
5. Build and push API image (D-07 tagging with short SHA + full SHA + latest):
```bash
IMAGE="${{ env.GAR_REGISTRY }}/api"
SHA="${GITHUB_SHA::7}"
docker build -f packages/api/Dockerfile \
  -t "${IMAGE}:${SHA}" \
  -t "${IMAGE}:${GITHUB_SHA}" \
  -t "${IMAGE}:latest" \
  .
docker push "${IMAGE}" --all-tags
```
6. Deploy to Cloud Run using `google-github-actions/deploy-cloudrun@v2`:
   - `service: seatkit-api`
   - `image: ${{ env.GAR_REGISTRY }}/api:${{ github.sha }}`
   - `region: ${{ env.REGION }}`
7. Print deployed URL from step output

**Job: deploy-web (D-16 — parallel, no `needs`):**

Same `if` condition pattern but checking for `web` or `both` in the workflow_dispatch case.

Steps identical to deploy-api except:
- Build command includes `--build-arg NEXT_PUBLIC_API_URL=https://api-staging.seatkit.dev` (D-20)
- Image path uses `/web` instead of `/api`
- deploy-cloudrun `service: seatkit-web`
- deploy-cloudrun `image: ${{ env.GAR_REGISTRY }}/web:${{ github.sha }}`

**Critical: Image tag consistency.**
The build step tags images with 3 tags: short SHA (`${GITHUB_SHA::7}`), full SHA (`${GITHUB_SHA}`), and `latest`. The deploy-cloudrun step references the full SHA via `${{ github.sha }}`. These are the same value (`GITHUB_SHA` env var in bash == `github.sha` in GitHub Actions expressions). This ensures the deployed image tag exactly matches a tag that was pushed.

Use `docker push "${IMAGE}" --all-tags` to push all three tags in one command instead of separate push commands per tag.

Both jobs need `permissions: contents: read` and `permissions: id-token: write`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions -> GCP | Service account key grants deploy permissions |
| GitHub Secrets -> workflow env | Encrypted secrets exposed as env vars in workflow |
| Docker build context -> image layers | Build must not leak secrets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.1-09 | Information Disclosure | GCP_SERVICE_ACCOUNT_KEY | mitigate | Stored as GitHub Actions encrypted secret; never logged; auth action handles credential file cleanup |
| T-05.1-10 | Spoofing | workflow_run trigger | mitigate | `if: github.event.workflow_run.conclusion == 'success'` prevents deploy on failed CI; `branches: [main]` prevents deploy from non-main branches |
| T-05.1-11 | Tampering | Docker image in transit | accept | Artifact Registry uses HTTPS; image digest verification happens automatically; acceptable for staging |
| T-05.1-12 | Denial of Service | Concurrent deploys | mitigate | `cancel-in-progress: false` prevents deploy cancellation; Cloud Run handles traffic migration atomically |
</threat_model>
