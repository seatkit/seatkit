### Pattern 2: next-themes ThemeProvider Setup (App Router)

**What:** Wrap the app with ThemeProvider in layout.tsx. Must be a client component wrapper.
**When to use:** Dark mode infrastructure task.

**Implementation:**
```typescript
// packages/web/src/components/theme-provider.tsx
'use client';

import { ThemeProvider as NextThemesProvider } from 'next-themes';
import type { ComponentProps } from 'react';

export function ThemeProvider({
  children,
  ...props
}: ComponentProps<typeof NextThemesProvider>) {
  return <NextThemesProvider {...props}>{children}</NextThemesProvider>;
}
```
[CITED: next-themes README + shadcn dark mode docs pattern]

**Critical:** The `<html>` tag in `layout.tsx` MUST have `suppressHydrationWarning` attribute. next-themes injects a synchronous script that modifies `<html>` before React hydrates, which causes React hydration warnings without this attribute. [VERIFIED: next-themes docs]

```typescript
// In layout.tsx
<html lang="en" suppressHydrationWarning>
  <body>
    <ThemeProvider
      attribute="class"
      defaultTheme="dark"
      enableSystem
      disableTransitionOnChange
      storageKey="seatkit-theme"
    >
      {/* ... */}
    </ThemeProvider>
  </body>
</html>
```

**Key props:**
- `attribute="class"` -- adds `.dark` class to `<html>` (matches Tailwind `darkMode: ['class']`)
- `defaultTheme="dark"` -- dark-mode-first per D-01
- `enableSystem` -- respects system preference
- `disableTransitionOnChange` -- prevents flash during theme switch
- `storageKey="seatkit-theme"` -- localStorage key per DESIGN.md

### Pattern 3: ThemeToggle Component

**What:** Icon button cycling system -> dark -> light
**When to use:** Placed in nav between wordmark and PresenceBadgeRow

```typescript
// packages/web/src/components/theme-toggle.tsx
'use client';

import { Monitor, Moon, Sun } from 'lucide-react';
import { useTheme } from 'next-themes';
import { useEffect, useState } from 'react';

export function ThemeToggle() {
  const { theme, setTheme } = useTheme();
  const [mounted, setMounted] = useState(false);

  // Avoid hydration mismatch - only render after mount
  useEffect(() => setMounted(true), []);
  if (!mounted) return <div className="w-8 h-8" />; // Placeholder

  const nextTheme = theme === 'system' ? 'dark' : theme === 'dark' ? 'light' : 'system';
  const Icon = theme === 'system' ? Monitor : theme === 'dark' ? Moon : Sun;
  const label = `Switch to ${nextTheme} mode`;

  return (
    <button
      type="button"
      onClick={() => setTheme(nextTheme)}
      aria-label={label}
      className="inline-flex items-center justify-center w-8 h-8 rounded-md hover:bg-muted focus:outline-none focus:ring-2 focus:ring-ring focus:ring-offset-2"
    >
      <Icon className="w-4 h-4" />
    </button>
  );
}
```

### Anti-Patterns to Avoid

- **Hardcoded Tailwind colors instead of CSS variable tokens:** The entire point of this phase is to eliminate `text-gray-600`, `bg-blue-600`, etc. and replace with `text-muted-foreground`, `bg-primary`, etc. The globals.css already defines complete light/dark token sets.
- **Animating virtualized rows:** `@tanstack/react-virtual` recycles DOM nodes -- `AnimatePresence` and `motion layout` will break if applied to virtual row containers. Timeline skeleton loading can animate, but the rendered grid must not.
- **Overriding shadcn component internals:** Sheet, Dialog, Tabs, Sonner come with Radix transitions. Do not apply additional motion animations on top of their built-in enter/exit transitions -- this causes double animation.
- **Missing `suppressHydrationWarning`:** Omitting this from `<html>` when using next-themes causes React hydration warnings in development and can cause SSR/client mismatch errors.
- **Using `useTheme()` during SSR:** next-themes theme value is `undefined` on the server. Always guard with a `mounted` state check to avoid hydration mismatches.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dark mode with SSR | Custom localStorage + script | next-themes | Handles SSR hydration, FOWT, system preference, class strategy -- 10+ edge cases |
| Slide-over drawer | Native `<dialog>` + CSS transforms | shadcn Sheet (Radix Dialog) | Focus trapping, ARIA semantics, portal rendering, keyboard navigation built in |
| Confirm dialog | Custom div overlay with manual focus | shadcn Dialog (Radix Dialog) | Radix handles focus trap, escape to close, backdrop click, screen reader announcements |
| Toast notifications | Custom fixed-position div + timer | Sonner (via shadcn) | Stacking, swipe-to-dismiss, action callbacks, accessible announcements, z-index management |
| Tab strip | Manual button array + state | shadcn Tabs (Radix Tabs) | Arrow key navigation, ARIA roles (tablist/tab/tabpanel), automatic panel association |
| Skeleton loading | Custom `animate-pulse` divs | shadcn Skeleton | Consistent pulse animation, composable width/height props, design token integration |
| Reduced motion detection | `window.matchMedia` polling | `useReducedMotion()` from motion | Hook handles SSR, subscription, and cleanup. Returns boolean. |

**Key insight:** Every shadcn component wraps a Radix primitive that implements WAI-ARIA Authoring Practices. This is hundreds of lines of accessibility code (focus management, keyboard navigation, screen reader announcements) that would need to be replicated if hand-built. The current codebase's native `<dialog>` and custom tab strip lack several of these behaviors.

## Common Pitfalls

### Pitfall 1: shadcn CLI Without components.json
**What goes wrong:** Running `npx shadcn add sheet` fails with "components.json not found"
**Why it happens:** The shadcn CLI was never initialized in this project -- no `components.json` exists
**How to avoid:** Run `npx shadcn init` first, or manually create `components.json`. Must be in the package root (`packages/web/components.json`), not project root.
**Warning signs:** CLI error about missing config

### Pitfall 2: cn() Utility Path Mismatch
**What goes wrong:** shadcn components import from `@/lib/utils` but `cn()` lives in `@seatkit/ui/lib/utils`
**Why it happens:** shadcn CLI generates import paths based on `components.json` aliases. The monorepo has `cn()` in the UI package.
**How to avoid:** Create `packages/web/src/lib/utils.ts` that either re-exports from `@seatkit/ui/lib/utils` or contains its own `cn()` (clsx + tailwind-merge). shadcn components will import from here.
**Warning signs:** TypeScript errors on first shadcn component import

### Pitfall 3: Flash of Wrong Theme (FOWT)
**What goes wrong:** Page flashes white (default) before switching to dark mode
**Why it happens:** React hydration happens after initial paint. Without a synchronous script, the `<html>` element doesn't have the `.dark` class during first paint.
**How to avoid:** next-themes handles this automatically when configured correctly -- it injects a synchronous `<script>` that reads localStorage before React hydrates. The key props are `attribute="class"` and `enableSystem`.
**Warning signs:** Visible white flash on page load in dark mode

### Pitfall 4: Hydration Mismatch with useTheme
**What goes wrong:** Server-rendered HTML doesn't match client (theme icon is wrong on first render)
**Why it happens:** `useTheme()` returns `undefined` on the server but `'dark'` on the client
**How to avoid:** Use a `mounted` state guard: only render the theme-dependent icon after `useEffect` sets `mounted = true`. Render a placeholder (same dimensions) during SSR.
**Warning signs:** React hydration warning about text content mismatch

### Pitfall 5: tailwindcss-animate Plugin Missing
**What goes wrong:** shadcn component animations (Sheet slide, Dialog fade) don't work -- components appear/disappear instantly
**Why it happens:** shadcn components use `tailwindcss-animate` utility classes (`animate-in`, `slide-in-from-right`, etc.)
**How to avoid:** Install `tailwindcss-animate` and add it to `tailwind.config.ts` plugins array
**Warning signs:** Components mount/unmount without transitions

### Pitfall 6: Status Color Tokens Not in Web Config
**What goes wrong:** `bg-status-confirmed`, `bg-status-pending` etc. classes don't apply
**Why it happens:** Status color tokens exist in `@seatkit/ui/tailwind.config.ts` but NOT in `@seatkit/web/tailwind.config.ts`
**How to avoid:** Copy the status color definitions from UI config to web config. The simplest approach -- they're 6 hardcoded HSL values that don't use CSS variables.
**Warning signs:** Status badges render with no background color

### Pitfall 7: Sheet Portal vs Existing Z-Index
**What goes wrong:** Sheet renders behind the nav or other elements
**Why it happens:** shadcn Sheet renders via Radix Portal (appended to `<body>`). If existing elements have high z-index values, the portal overlay may not stack correctly.
**How to avoid:** The nav is `z-30`, the drawer was `z-50`. shadcn Sheet defaults to a high z-index via its overlay. Verify stacking after migration.
**Warning signs:** Drawer content hidden behind nav or other fixed elements

### Pitfall 8: Sonner Replacing UndoToast Undo Functionality
**What goes wrong:** The "Undo" action in Sonner fires the callback but the undo mutation doesn't execute
**Why it happens:** The current UndoToast reads from `useReservationUndoStore()` which holds the undo snapshot. Sonner's action callback fires independently of React state.
**How to avoid:** Wire the Sonner action callback to call a function that reads from the undo store at invocation time (not at toast creation time). Use a ref or store selector.
**Warning signs:** Clicking "Undo" does nothing or throws

## Open Questions (RESOLVED)

1. **cn() utility placement** -- RESOLVED: standalone copy in web package (Plan 01 Task 1)
   - What we know: `cn()` (clsx + tailwind-merge) exists in `@seatkit/ui/lib/utils`. shadcn CLI expects it at `@/lib/utils`.
   - What's unclear: Whether to re-export from UI package or create a standalone copy in web package
   - Recommendation: Create `packages/web/src/lib/utils.ts` with a standalone `cn()` -- simpler, avoids cross-package import in UI primitive layer

2. **ConfirmDialog migration scope** -- RESOLVED: new Dialog-based component in web package (Plan 03 Task 2)
   - What we know: ConfirmDialog lives in `@seatkit/ui` package. Migration to shadcn Dialog means either (a) replace in-place in UI package, or (b) create new Dialog-based component in web package.
   - What's unclear: Whether `@seatkit/ui` should take a dependency on Radix Dialog
   - Recommendation: Migrate ConfirmDialog to use shadcn Dialog in the web package (create `web/src/components/confirm-dialog.tsx`), and update imports in settings pages. The old UI package version can remain for backward compatibility but new code uses the web version.

3. **Sonner Toaster placement with ThemeProvider** -- RESOLVED: Toaster inside ThemeProvider with explicit CSS var style mapping (Plan 01 Task 2)
   - What we know: Sonner needs a `<Toaster>` component rendered once in the app
   - What's unclear: Whether Sonner reads the `.dark` class automatically for theme-aware styling or needs explicit CSS variable wiring
   - Recommendation: Place `<Toaster>` inside `ThemeProvider` wrapper. Use `toastOptions.style` to map to CSS variables explicitly.
